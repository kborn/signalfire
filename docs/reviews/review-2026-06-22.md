## Visual Design — 6.0/10

The palette and typography are the site's strongest assets. Dark navy + amber with Playfair Display headings gives FYF genuine visual identity and gravitas that most portfolio projects never achieve. Unfortunately, almost everything built on top of that foundation squanders it.

### Issues

**The logo is embarrassing.**
"FYF" in amber text at 14px is not a brand mark. It's a placeholder that shipped. A portfolio reviewer's first impression of the brand is three capital letters with no icon, no treatment, no weight. This is the most visible signal that the site is unfinished. Either invest in a real wordmark (even a typographic one with proper sizing and letter-spacing) or drop the abbreviation entirely and use "Find Your Fight" in Playfair Display at a size that commands the navbar.

**The hero buries the value proposition under three paragraphs.**
The home hero asks users to read 200+ words of copy before reaching the CTA. "Find yours →" appears after two full paragraphs of motivational text. Most users will have already scrolled past it or tuned out. The lead paragraph is actually quite good — stop there. Cut everything that follows in the hero and move it to About. The CTA text "Find yours →" also fails: find your _what_, exactly? This is the one moment to be explicit. "Find your issue" or "Browse issues →" does actual work.

**The "Three Steps" section looks like a brochure, not a product.**
Three identical gray cards with step numbers and generic descriptions. There's no visual differentiation, no sense of forward motion, no imagery. The cards are clickable but they don't visually communicate that. A first-time visitor reading "Step 1: Go deep on one issue" could be looking at a marketing one-pager that leads nowhere.

**Action detail page has clinical whitespace.**
The action detail (screenshot 04) is mostly empty space. A title, a thin separator, a one-sentence summary, a separator, a CTA button, then another separator, then metadata, then more content. That's five visual breaks in a 600px scroll. The page reads like a form, not a content destination. The metadata block — Action Type, Published, Updated — sits visually between the CTA and the description, which is the worst possible placement for it.

**The `bg-motif.png` disappears on most pages.**
On the homepage it's compelling. On collection pages it fades into the dark background and becomes invisible noise. If the motif is intentional (it is, per CLAUDE.md), it needs enough opacity to read, or it should only appear on hero sections where it has breathing room.

**The Issues page icon badges are too small.**
The colored emoji-style icons in the topic cards (screenshot 02) are approximately 20×20px at actual screen size. They barely register and don't differentiate the cards from each other meaningfully. Either make them larger or remove them — right now they occupy visual space without earning it.

## UX / Product — 5.5/10

The platform's stated purpose is bridging the gap between knowing and acting. The site describes that journey in three places (hero, How It Works, About) but never actually delivers it. A user who clicks "Find yours →" gets scrolled to a list of issue links on the same page. That is not a product moment.

### Issues

**The core Learn → Decide → Act journey is explained but never guided.**
"Step 1: Go deep on one issue" links to `/issues` — a browse list. "Step 2: Read what matters" links to `/articles` — another browse list. "Step 3: Do one concrete thing" links to `/actions` — another browse list. The site describes a journey and then immediately asks the user to navigate it entirely on their own. This is the biggest product gap. An issue detail page that chains together: here's the context (articles), here are the actions you can take, here are nearby events — that's the journey. The homepage just needs to point at _that page_ as the destination, not three separate browse lists.

**The events filter form dominates before a single event is visible.**
Screenshot 03 shows: page header, descriptive copy, a helper text note about demo data, then a filter form with state/territory selector, city input, date range pickers, and a topic selector — all before a single event card appears. That's 400px of controls before you see any content. Users scan first, filter second. The filter should collapse by default or be secondary to the event list.

**Action detail metadata is sorted by internal interest, not user interest.**
On the action detail page: title → summary → CTA → Action Type, Published date, Updated date → description → related issues. The user's mental model is: title → what is this → what do I do → background → tags/meta. The actual order is: title → what is this → what do I do → **administrative metadata nobody asked for** → background. Published and Updated dates add zero value to a user deciding whether to take a civic action. Strip them from the public detail page entirely or relegate them to a collapsed meta section at the bottom.

**"Take Action on example.org →" is a trust problem.**
In demo mode, the CTA on action pages reads "Take Action on example.org →". This is sample data — understood. But for a portfolio demo, this is what a reviewer sees. The domain label is meant to build trust ("you're going to a real external site"). "example.org" does the opposite. Either seed the demo data with realistic external URLs (actionnetwork.org, change.org, vote.org) or suppress the domain label when the URL is an obvious placeholder.

**The submission flow lacks continuity.**
After submitting content, does anything happen? Is there a confirmation state, an expected timeline, a way to check status? The admin submission-review screenshot (05) shows a clean workflow on the backend, but there's no indication of what the submitter experiences after hitting submit. This is a two-sided platform; right now only one side has a coherent experience.

**No editorial voice on collection pages.**
Actions list, articles list, events list: each is a grid of cards with no editorial hierarchy. Nothing is featured, highlighted, or recommended. Every item has identical visual weight. The platform's value is supposed to be curation — that curation should be visible, not just implied by the fact that everything passed moderation.

**"Browse all issues →" and "Browse more actions →" are ghost links.**
Both appear as small text links at the bottom of sections. In the site's visual language, the primary CTA button is amber and outlined. These text CTAs are styled identically to paragraph text. Users glancing at the page will not register these as interactive.

## Engineering — 7.5/10

The codebase is cleaner than the feature count would suggest. Server components are used correctly, the API contracts package is a mature architectural choice, and there are no visible hacks or shortcuts in the routing logic. The bones are good.

### Issues

**The custom CSS class system is accumulating debt.**
The pages.css file is 573 lines of page-specific class names: `heroPosterRoll`, `heroPosterIssueLink`, `homeJourneyCard`, `discoveryEmptyState`, `detailHero`, `detailContributeNudge`. These are one-of-a-kind selectors with no reuse and no compositional system behind them. The layout.css defines good design tokens, but the page-level CSS doesn't consistently build on those tokens — it creates new spacing, sizing, and color values inline. Two more phases and this file becomes the place where going fast means breaking things. Either adopt Tailwind (which has been rejected per decisions.md presumably) or invest in a real utility layer on top of the existing token system before this hardens.

**The `variant` prop pattern on summary components is a smell.**
`ActionSummary`, `ArticleSummary`, and similar components accept a `variant` prop that switches between `'collection'` and `'related'` display modes — different class names, different heading tags, different sizes. This is a hidden abstraction tax: you can't understand what a component renders without tracing all variant branches. Two separate components (`ActionSummaryCard` and `ActionSummaryRelated`) would be more honest and easier to refactor independently.

**No loading or error states are visible in the public UI.**
Collection skeleton components exist (`collection-skeleton.tsx`) but there's no evidence from the screenshots or route structure that streaming suspense boundaries are wired up between the header/navigation and the dynamic content sections. On a slow API response, users will see either a blank page or a stalled layout. For a demo site this matters — reviewers will interact with it across variable network conditions.

**`await connection()` in every page is a manual footgun.**
Every page server component manually calls `await connection()` to opt out of static rendering. This is correct behavior but it's a copy-paste pattern distributed across 15+ pages. If a new page forgets it (and one eventually will), it silently renders stale data with no indication. A wrapper layout that forces dynamic rendering at the route group level would eliminate the entire class of mistake.

## What Is Actually Good

- **The palette is excellent.** Dark navy + amber is genuinely distinctive in the civic tech space. It doesn't look like an NGO's SharePoint and it doesn't look like a startup's Figma export. It has a voice.
- **Playfair Display at large sizes is doing real work.** The `h1` on the homepage and action detail pages lands well. This was the right typographic choice.
- **The copy throughout is strong.** Not a single "Lorem ipsum" visible in public-facing copy. The About page is better written than most real products. The voice is consistent.
- **The topic-based color accent system is a good idea.** CSS custom property `--topic-accent` threaded through breadcrumbs, cards, and related items is architecturally sound. It scales to any number of topics without new CSS.
- **The API contract separation is mature.** Having `@signal-fire/api-contracts` as a shared package means the web app and API stay in sync at the type level. This is the right call for a project that will grow.
- **The admin submission review workflow is functional.** It's plain but it does the job. The moderation metadata layout (screenshot 05) is readable and complete.
- **The events filter chip design (screenshot 03)** — the topic pills row (`Democracy`, `Consumer Activism`, `Climate`, etc.) is the best UI widget on the site. Clean, scannable, responsive.
