# Site Review — Find Your Fight

Harsh review of the public site, admin workspace, and codebase as of Phase 13 close. Intended as a pre-deployment quality gate.

---

## Scores

| Dimension     | Score        |
| ------------- | ------------ |
| Visual Design | 6 / 10       |
| UX / Product  | 5 / 10       |
| Engineering   | 6 / 10       |
| **Overall**   | **5.5 / 10** |

---

## Visual Design — 6 / 10

The palette is intentional and the dark navy + gold combination has personality. The motif background, hero image treatment, and collection card left-border accent all read as deliberate choices rather than defaults. That's the ceiling. Here's everything that pulls it down:

### Issues

**No footer.**
The public shell has no footer at all. Long pages just end. There's no secondary navigation, no copyright line, no "back to top" affordance. On detail pages a reader finishes an article and has nowhere to go except to use the browser back button or the sticky header. This reads as unfinished.

**The article detail column order is wrong.**
`detailContent--article` is a two-column grid: `minmax(0, 280px) minmax(0, 1fr)`. Column 1 is the metadata sidebar, column 2 is the article body. This means readers encounter the author and date metadata before the actual content. On all published media — newspapers, magazines, web articles — the content leads and the metadata is secondary. The sidebar needs to move right.

**Related sections in article detail are crammed into the wrong column.**
The `relatedSection` divs inside `detailContent--article` have no `grid-column` specified. They flow into the 2-column grid as children 3 and 4, landing in column 1 (280px) and column 2 respectively. The "Explore This Issue" related topics section likely renders at 280px wide — very narrow for a list of topic cards. These sections should `grid-column: 1 / -1` to span the full width.

**Hardcoded hex values undermine the design system.**
The CSS variables system (`--color-text-primary`, `--color-brand-accent`, etc.) is good, but numerous rules bypass it:

- `collectionItemSummary { color: #2d2d2d }` — only visible on light backgrounds; the dark shell override in `.publicShell .collectionItemSummary` saves it but this is fragile.
- `metaText { color: #444444 }`, `eventMeta { color: #444444 }`, `detailMetaGroup { color: #555555 }` — all hardcoded.
- `submissionLabel { color: #171717 }` in admin context is overridden everywhere by editor form classes. Remove the base rule and set the correct value in each context.
- `.secondaryCTA { border: 1px solid #171717 }` — hardcoded light-mode color that requires `.publicShell .secondaryCTA` to override it for the dark theme. The base component shouldn't assume light mode.

**`.secondaryCTA` has `margin-right: 10px` baked in.**
This is a layout concern in the component style. It causes uneven spacing when multiple secondary CTAs appear next to each other and needs to be removed from the component and handled in the parent layout.

**"Submit Content" pill in the sticky header on every page.**
The pill CTA makes sense on the homepage and submit pages. On an article detail page, a topic detail page, or an event page mid-read, it is noise. Consider limiting this to the home and about pages, or at minimum reducing it to a plain text link in the nav.

## UX / Product — 5 / 10

The concept — focus → learn → act — is clear and the homepage communicates it reasonably well. But several execution problems undercut the experience:

### Issues

**The events page contradicts its own design decision.**
The `decisions.md` entry dated 2026-06-13 says: _"The public /events page should not render a generic Event list before the user supplies a meaningful filter set."_

The current implementation doesn't do this. `getContents(params)` is called unconditionally at line 138 regardless of whether `hasLocationFilter` is true. The `!hasLocationFilter` block shows a guidance message ("Upcoming events are already on the page. Add a state, city, or issue filter...") but the events list still renders below it.

This is the worst kind of UX problem: the copy promises one thing and the implementation does another. A reviewer clicking into Events sees both the guidance panel AND a full event list simultaneously. Pick one direction and commit: either show events by default (and remove the guidance), or gate on a filter (and don't render results until one is set).

**`/demo` page is missing.**
`decisions.md` (2026-06-16) explicitly calls for a public `/demo` page as the canonical path to admin access instructions. The git status shows `apps/web/src/app/(public)/demo/page.tsx` as `AD` — added to the staging index but deleted from disk. The header links "Admin Demo" directly to `/admin` (the login gate), not to a `/demo` explainer. First-time reviewers who aren't sure whether admin access is open or what the demo credentials are have nowhere to look except the repository README. This is a direct failure of the stated design goal of making the site self-contained for portfolio review.

**"Browse Events For This Issue" always appears on topic pages.**
The CTA on topic detail pages (`/topics/[slug]`) always renders, regardless of whether there are any events for that topic. If a reviewer clicks "Browse Events For This Issue" and gets an empty state, that's a dead end. Either conditionally render the CTA (not trivial without a backend check) or change the copy to not imply events exist.

**"Take Action" in the nav but "Actions" on the page.**
The nav label is "Take Action." The page title is "Actions." The hero section label is "Browse." These should be consistent. Pick "Actions" everywhere or pick "Take Action" everywhere — not both.

**No loading/transition feedback on filter changes.**
When the topic selector or event filters change, the page navigates and re-renders server-side. There is no pending state indicator. Fast connections won't notice, but on slower connections the page appears frozen for a moment with no indication that something is happening.

**About page Step 3 uses a mixed link pattern.**
Steps 1 and 2 on the About page use `<Link>` directly as the `.about-step` card, making the whole card clickable. Step 3 wraps a `<Link>` inside a `<section>` with CTAs outside the link. The interaction model is inconsistent — hover behavior and click targets differ between steps in the same visual component.

**`p { max-width: 70ch }` is applied globally.**
This aggressive global rule caps all paragraph widths at 70 characters. It affects admin panels, form labels, error messages, and everywhere else a `<p>` appears. Many of these contexts need uncapped width and require explicit overrides (`max-width: none`). This is a design decision that shouldn't be expressed as a universal reset.

## Engineering — 6 / 10

The structure is solid: NestJS modules with clear ownership, Prisma ORM, Next.js App Router, good test coverage across controller/service/repository layers. The foundational decisions are sound. The execution-level problems are real though.

### Issues

**`globals.css` is 3,986 lines.**
Everything — public layout, admin panels, editor forms, event filters, submission forms, animations, responsive breakpoints — lives in a single file. It's completely unmaintainable. Adding a new feature requires grepping a 4k-line file to find where something is defined, then hoping the cascade doesn't produce unexpected side effects. This should be broken into component-level CSS modules, at minimum by domain (public.css, admin.css, forms.css).

**Three editor forms duplicate the same ~60 lines of CSS each.**
`actionEditorForm`, `eventEditorForm`, and `articleEditorForm` each define identical override blocks for `.adminPanel`, `.adminPanelHeader h2/h3`, `.submissionLabel`, `.submissionControl`, `.submissionTextarea`, `.submissionControl:focus`, `.submissionControl:disabled`, `.submissionCheckboxGroup`, `.adminTopicGrid`, `.submissionCheckboxOption`, `.submissionCheckboxOption input`, and footer/success banner variants. That's approximately 180 lines of tripled CSS that should be a single `.editorForm` class.

**API defaults to port 3000 — same as Next.js.**
`main.ts` line 28: `await app.listen(process.env.PORT ?? 3000)`. The Next.js dev server also defaults to port 3000. Running both locally without setting `PORT` causes a silent conflict where one process fails to bind. This should default to 3001 for the API (or the env example should make this explicit and required).

**`EventListPageProps` type is duplicated.**
The same type is defined identically in `events/page.tsx` and `events/_components/event-filters.tsx`. They should share a single type from a common location.

**`getContents` async call inside JSX.**
`events/page.tsx` line 138: `{await getContents(params)}`. While valid in Next.js 15 server components, calling an async function inline in JSX is non-standard and harder to follow. The data fetching should happen at the page level alongside the `getTopicsList` call.

**`resolveDateWindow` uses UTC month math that can produce wrong local dates.**
The function adds 3 months to a UTC date using `date.setUTCMonth(date.getUTCMonth() + 3)`. For end-of-month dates this can produce unexpected results (e.g. adding 3 months to Jan 31 produces May 2, not Apr 30). The time module should handle this more carefully.

**No `rel="noopener noreferrer"` on external links.**
The codebase uses Next.js `<Link>` for internal navigation (correct) but any external links rendered via `<a>` in markdown content or admin views should carry `rel="noopener noreferrer"`. Worth auditing the markdown renderer.

**`topicSelector a[aria-current='page']`** is semantically wrong.
The topic filter bar uses `aria-current="page"` to mark the active topic. `aria-current="page"` means the link points to the current page. A topic filter is not a page — it's a selected state. This should be `aria-selected="true"` or `aria-pressed="true"` depending on the interaction model.

**The demo banner animation hardcodes `2.5s` enter duration.**
`animation: demoBannerEnter 2.5s cubic-bezier(0, 0, 0.2, 1)` — 2.5 seconds is a very long entrance for a UI element that appears on every public page load. Users who have dismissed it once have no way to prevent it from animating again on next visit (session storage is used for dismissal but there's no persistence across sessions).

## Priority Order for Phase 13 Remaining Work

1. **Critical — blocks portfolio readiness:**
   - Create the `/demo` page (it's missing entirely)
   - Fix the events page to resolve the filter-led vs. show-all contradiction
   - Fix article detail: move metadata sidebar right, fix related section column span

2. **High — visible to any reviewer:**
   - Replace "FYF" header with "Find Your Fight" or at least an interpretable wordmark
   - Add a footer with minimal navigation
   - Fix API default port (3000 conflicts with Next.js)

3. **Medium — engineering credibility:**
   - Break `globals.css` into at minimum 3 files (public, admin, forms)
   - Merge the three editor form CSS blocks into one `.editorForm` shared class
   - Replace hardcoded hex values with CSS variables throughout

4. **Lower — polish:**
   - Remove `margin-right: 10px` from `.secondaryCTA`
   - Fix `aria-current="page"` → `aria-pressed` on topic selector
   - Scope the `p { max-width: 70ch }` rule to avoid admin/form collateral damage
   - Make `demoBannerEnter` animation duration shorter (0.4s–0.6s is standard)
   - Conditionalize "Browse Events For This Issue" CTA on topic pages
