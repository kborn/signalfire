## Baseline Findings (2026-06-17)

### Visual Design — 4/10

**The logo is a placeholder that never got replaced.** `· FYF ·` with flanking CSS pseudoelement dots reads as typography experimentation that was never resolved. The dots are `::before`/`::after` on `.site-wordmark` in `globals.css:150–158`. For a civic action platform with a clear brand name, this reads as unfinished.

**The hero hierarchy is inverted.** "FIND YOUR FIGHT" in outlined type has energy, but the outlined-only treatment on "FIGHT" competes with "FIND YOUR" instead of being subordinate. The `choose · learn · act` subline is barely readable against the dark background. The hero section is mostly empty space with a paragraph and two buttons crammed into the lower-left.

**The Issues index is brutal to browse.** Ten full-width cards, each with "ISSUE" label, title, description. Zero visual differentiation between topics — no imagery, no iconography, no color coding per issue, no metadata beyond text. Cards are dressed-up list items.

**The Topic detail page has spec-note copy.** "STEP 2 - READ ENOUGH TO ACT" with "Read background and explainers for this issue." tells the user exactly what they're already doing. Nothing visually distinguishes "Climate" from "Democracy" beyond the description text.

**The article detail page renders the title twice.** `articles/[slug]/page.tsx:35` renders `<h1>{article.title}</h1>`. The markdown body content starts with its own H2 that duplicates the title. Visible in `docs/screenshots/04-article-detail.png`.

**The Events filter `<select>` has broken dark-theme styling.** `globals.css` applies `color-scheme: dark` to `input[type='date']` (line 1494) and `input[type='datetime-local']` (line 1486) but not to `<select>` elements. The state dropdown renders with browser-native light background, visually broken against the dark page.

**The admin area is a jarring context switch.** Public site: dark, branded, high-contrast. Admin workspace: light gray, `#eef2f5` background, no brand. The admin floats as a rounded card (`border-radius: 24px`) on the dark page background (`#101820`), creating a floating-card-on-black effect with empty dark space below.

**Screenshot `docs/screenshots/08-submission-review-decision.png` is a 404 page.** Filed a broken Not Found as a product screenshot.

### UX / Product — 4/10

**The Events page is a dead wall on arrival.** `events/page.tsx:39–41` — `canRequest()` returns `Boolean(props.region?.trim())`, meaning no region selected → no results, just a prompt card. No default browse mode, no national view, no "most recent events" fallback. Biggest user journey failure in the product.

**"Ways to Act" is a weak nav label.** `(public)/layout.tsx:29` — Issues, Articles, Events, About are all one word. "Ways to Act" is three. Pick one.

**The article detail page is a dead end.** You read an article and there is no breadcrumb back to the topic, no next/prev article, no "more like this." The article already has `article.topics` available (line 61) and does nothing with it for navigation.

**No keyword search.** A content discovery platform with no full-text search. Topics taxonomy is the only discovery affordance and requires knowing which bucket you're in.

**The content graph is over-engineered for what's visible.** Six many-to-many join tables. With thin seed data, every page renders the same card layout with different text. The graph exists in the database but you can't feel it as a user.

**Event times display UTC for local events.** "Wed, Jul 8, 2026 at 2:00 PM UTC" for a Philadelphia event. Events are inherently local — UTC display is confusing and wrong for attendees.

**Topics are not admin-manageable.** Articles, Actions, and Events can be created/edited in the admin workspace. Adding a new Issue category requires a developer with database access.

### Engineering — 6.5/10

**`noImplicitAny: false` in `tsconfig.base.json:9`.** For a TypeScript-first monorepo with shared contracts as a selling point, this undermines the whole approach. Also `strictBindCallApply: false` on line 10.

**`dist/` checked into version control.** `packages/admin-auth-shared/dist/` contains `index.d.ts`, `index.js`, `tsconfig.tsbuildinfo`. Build artifacts don't belong in the repo.

**`submission_type.ts` uses snake_case.** Every other file in `packages/api-contracts/` uses kebab-case. Inconsistent in the canonical type package.

**No rate limiting on the public submission endpoint.** `apps/api/src/submission/submission.controller.ts:12` — bare `@Post()` with no throttle guard. Moderation queue can be flooded by a bot.

**`force-dynamic` on every content page.** `articles/[slug]/page.tsx:9`, `topics/[slug]/page.tsx:4`, `actions/[slug]/page.tsx`, and others all declare `export const dynamic = 'force-dynamic'`. A moderation-gated content platform is the canonical ISR use case. Content only changes when an admin publishes. Right now every page load hits the database with no caching. This is a meaningful architectural mistake.

**3,681-line monolithic CSS with an inconsistent custom property system.** `globals.css` defines a clean token system at lines 1–22 then ignores it throughout:

- `line 914`: `.collectionItemSummary { color: #2d2d2d; }` (not a token)
- `line 1007`: `.eventMeta { color: #444444; }` (not a token)
- `line 1125`: `.detailMetaGroup { color: #555555; }` (not a token)
- `line 1272`: `.submissionLabel { color: #171717; }` (not a token)
- `line 1277`: `.submissionControl { border: 1.5px solid #9ca3af; background: #ffffff; }` (not tokens)
  ~20+ hardcoded hex values that should be token references. No CSS modules, no component scoping — styles qualified by ancestor classes (`.publicShell .collectionItem`) create tight coupling between component appearance and DOM context.

**~12 Google fonts loaded from an abandoned experiment.** `.next/dev/static/` shows Alegreya, Archivo Black, Arvo, Bitter, Instrument Serif, Inter (×2), Lora, Merriweather, Monoton, Source Serif 4 (×2), Zilla Slab. The deployed CSS uses `Georgia, 'Times New Roman', serif`. These are almost certainly dead imports in `apps/web/src/app/layout.tsx` from font selection experiments never cleaned up.

**`.secondaryCTA` has `margin-right: 10px` on the component itself.** `globals.css:799` — spacing between siblings belongs to the layout, not the component. Breaks composability.

The moderation workflow (submit → queue → review → normalize → approve/reject) is well thought-out and fully implemented. The NestJS module structure is disciplined. Testcontainers integration test setup is above average for a portfolio project. System architecture docs are thorough and maintained.

## Round 2 Fix List (Moves Post-Fix Scores Toward 8.5)

These are the fixes that make the engineering and design ceiling higher. Round 1 removes defects; Round 2 demonstrates craft.

### Engineering — Required to reach 9.0

**FIX-13 — Replace `force-dynamic` with ISR on all content pages** (highest impact)

- Files: `apps/web/src/app/(public)/articles/[slug]/page.tsx:9`, `topics/[slug]/page.tsx:4`, `actions/[slug]/page.tsx`, and all other public content pages with `export const dynamic = 'force-dynamic'`
- Replace with `export const revalidate = 300` (5-minute cache) as a baseline
- Better: implement on-demand revalidation — when an admin publishes or edits content, call `revalidatePath()` or `revalidateTag()` from the admin API route handler. This is the architecturally correct pattern for a moderation-gated content platform and the most impressive engineering signal in the codebase.

**FIX-14 — Enforce CSS token usage, eliminate hardcoded hex values**

- File: `apps/web/src/app/globals.css`
- Audit all bare hex color values and replace with the existing custom properties from `:root`. Key ones:
  - `line 914`: `#2d2d2d` → `var(--color-text-muted)` or a new `--color-text-secondary` token
  - `line 1007`: `#444444` → token
  - `line 1125`: `#555555` → token
  - `line 1272`: `#171717` → token
  - `line 1277–1279`: `#9ca3af`, `#ffffff`, `#171717` → tokens
- Optional: add an ESLint or Stylelint rule to ban bare hex in CSS outside of `:root`.

**FIX-15 — Audit and strip dead Google font imports**

- File: `apps/web/src/app/layout.tsx`
- Find all `next/font/google` imports. Remove any font not actively referenced by a CSS `font-family` declaration. The bundle currently loads ~12 fonts; the CSS uses one display font (Georgia fallback). Strip the experiment artifacts.

**FIX-16 — Fix `.secondaryCTA` component margin**

- File: `apps/web/src/app/globals.css:799`
- Remove `margin-right: 10px` from `.secondaryCTA`. Move spacing responsibility to layout containers (`.ctaRow` already uses `gap: var(--space-2)` which handles this correctly).

**FIX-17 — Event timezone: store and display local time**

- File: `apps/api/prisma/schema.prisma` — `Event` model
- Add a `timezone` field (e.g. IANA timezone string like `"America/New_York"`) to the `Event` and `Submission` models
- Update the event display components to render times in local timezone using `Intl.DateTimeFormat` with the stored timezone, not UTC

### Visual Design — Required to reach 8.0

**FIX-18 — Issues index: add visual differentiation between topics**

- File: `apps/web/src/app/(public)/topics/page.tsx` and `components/topic-summary.tsx`
- Every topic card currently looks identical. Options (pick one or combine):
  - Assign a distinct accent color per topic stored in the `Topic` model and surface it as a CSS custom property on each card
  - Add an icon or emoji per topic rendered as `collectionItemEyebrow`
  - Use a 2-column grid layout on wider viewports instead of a single column stack

**FIX-19 — Replace the `· FYF ·` wordmark**

- File: `apps/web/src/app/(public)/layout.tsx:21`, `globals.css:131–158`
- The current mark is CSS-generated dots flanking "FYF". Either commit to a proper SVG wordmark or make "Find Your Fight" the full logotype in the header instead of the abbreviation.

**FIX-20 — Hero `choose · learn · act` readability**

- File: `apps/web/public/hero.png` (image-based) or the hero section in `page.tsx`
- The subline is barely readable at current contrast. If it's baked into the hero image, update the image. If it's CSS text, verify contrast ratio meets WCAG AA (4.5:1 minimum).

### UX — Required to reach 8.5

**FIX-21 — Add keyword search**

- A content discovery platform needs full-text search. Minimum viable: add a search endpoint on the API using Prisma's `contains` filter across title/summary/content for Articles and Actions. Surface it as a search input in the header or a `/search` route.

**FIX-22 — Admin: make Topics manageable without a developer**

- Currently adding a new Issue category requires Prisma seed or direct DB access. Add a Topics CRUD section to the admin workspace following the same pattern as Articles/Actions/Events.
