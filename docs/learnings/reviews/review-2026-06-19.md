## Visual Design — 6.0/10

The homepage hero is genuinely strong and the typography choice (Playfair Display) is correct. The per-topic accent color system shows design intent. Everything else ranges from acceptable to embarrassing. One of the five portfolio screenshots is a 404 page.

### Issues

**Screenshot #5 is a broken 404**
`docs/screenshots/05-submission-review.png` renders a bare "Not Found" page with no layout, no header, no shell — just large white type on a black background. This is the admin submission review screen, presumably broken because the route required auth that wasn't provided when the screenshot was taken. Phase 13.6 had an explicit task to reduce screenshots to 5 focused portfolio captures and update them. One of those five is a 404. There is no softer way to say this: if a recruiter opens the README and sees this, the review is over.

**The grain motif is invisible**
`pages.css:24-31` — the `motifPage::before` generates a fixed-position Perlin noise texture at 3.5% opacity with `mix-blend-mode: overlay`. At that opacity, on the dark background, it is undetectable without zooming in to inspect the CSS. It either needs to be 8–10% opacity and actually visible, or removed. Right now it communicates "we tried something" without communicating anything visually.

**The bg-motif is too faint to read as intentional**
The `heroPoster::after` adds `/bg-motif.png` at 11% opacity in the top-right corner of the hero. On a dark background at that opacity, the reviewer has to squint to notice it. The intention is texture and brand identity. The result looks like an artifact or a compression ghost. Double the opacity or drop it.

**Action detail page is visually anemic**
Screenshot 04 shows an action detail with a title, three meta blocks (Action Type, Published, Updated) with their values, a description paragraph, and three related topics at the bottom. There is no primary CTA button visible even though the code shows a `Take Action →` button gated on `externalUrl`. Looking at the actual page structure, this means the screenshot was captured for an action without an external URL, making the primary purpose of an action page invisible. Whether that's a seed data problem or a screenshot selection problem, the output is the same: the most important screenshot of user-facing product value shows a page with no call to action.

**Visual system breaks down past the homepage**
The homepage has a considered visual hierarchy: large display type, issue roll with left-border hover accents, clean section breaks. The Issues grid uses a 2-column card layout with topic-specific left borders and icons. The article/action/event detail pages are structurally similar but feel assembled rather than designed. The meta blocks (Action Type, Published, Updated) render as small-caps labels above plain text with no visual grouping that would help a reader scan. The "Related Topics" section at the bottom of the action detail uses a left-border list style that looks like a code diff more than a navigation aid.

**Demo indicator redundancy**
There is a "Demo" pill badge in the header next to the wordmark AND a dismissible demo banner below the header. Both are visible simultaneously on every public page in demo mode. One of these communicates the demo context; two creates visual noise that competes with the actual content. Pick one.

**The nav is overfull**
At desktop width the nav packs: Issues · Articles · Actions · Events · [search icon] · About · Admin Demo · Submit. The search icon breaks the text-label pattern without explanation. "Admin Demo" is styled in brand gold, drawing the eye before "Submit" despite Submit being the primary action for visitors. The visual hierarchy of the header prioritizes the demo affordance over the actual product CTA.

## UX / Product — 5.5/10

The product concept is clear and the copy is honest. The implementation of that concept has structural problems that prevent the Learn → Decide → Act journey from actually working end to end.

### Issues

**The homepage issues list is hardcoded**
`apps/web/src/app/(public)/page.tsx:4-12` — the ISSUES constant is a static array of slugs and labels. The homepage does not call the topics API. Now that Topic CRUD exists in admin, a new topic created through the admin interface will appear in Issues, Articles, Actions, and Events discovery — but not on the homepage. A deleted or renamed topic will persist on the homepage. This is the worst kind of inconsistency: silently wrong, with no error.

**Events are architecturally isolated from the journey**
The stated product loop is Learn → Decide → Act. Events don't fit this cleanly and it shows. Events have an entirely different UX model (filter-first vs browse-first), don't appear inline on Issue/Article/Action detail pages, and are accessible only through a separate nav destination. The connection between an issue (Democracy) and a relevant event (a town hall about voting rights) is not surfaced on the issue detail page — only a generic "Find Events" CTA appears. A first-time visitor following the Issue → Read → Act path will never encounter Events unless they proactively navigate there.

**The "Learn → Decide → Act" journey is labeled but not experienced**
Issue detail pages have "Step 1 — Read" and "Step 2 — Act" section headers. These read as documentation about the journey, not as experience design. A user who clicks into an issue sees a description, some article links under "Step 1 — Read," some action links under "Step 2 — Act." The steps are just visual section dividers. There is no progressive disclosure, no "you're here" orientation, no forward momentum that distinguishes this from a standard blog category page with renamed sections.

**Navigation taxonomy creates duplicate pathways without hierarchy**
Issues, Articles, and Actions are all top-level nav items. But Articles and Actions are also nested under Issues. This creates two mental models simultaneously: "Articles exist on their own" and "Articles are part of an Issue." This isn't resolved for the user anywhere. The About page, homepage copy, and issue detail structure each describe the relationship slightly differently. A visitor who lands on `/articles` directly has no framing about how those articles relate to issues. A visitor who lands on an issue detail doesn't know that `/articles` shows all articles across all issues.

**No primary CTA visible on action detail (critical for the product)**
If the seeded demo action doesn't have an `externalUrl`, the action detail page shows: a title, metadata (type, dates), a description, and related links. There is no button. The product is a civic action platform and the action detail — the page that exists specifically to enable action — has no action affordance visible above the fold unless the seed data happens to include an external URL. This is a core user flow that must work by default, not only when a data field is populated.

**Submission flow is not placed at moments of high intent**
The submission CTA ("Submit Content") is in the header nav and on the homepage contribute section. It is not surfaced on article detail pages ("Know something related?"), event pages ("Did we miss an event?"), or issue detail pages ("Have a resource to share?"). Contribution is promoted at arrival and departure, but not in the middle of the journey where intent to contribute is highest.

**The /demo page as admin entry point adds friction**
In demo mode, "Admin Demo" in the header routes to `/demo` which then explains the admin workspace and presumably links to `/admin`. This is one click further than routing directly to admin. The justification (give reviewers context before they hit the admin interface) is valid for a public portfolio but creates a dead-end page that is almost certainly confusing to anyone who clicks "Admin Demo" expecting to see admin.

## Engineering — 6.0/10

The underlying architecture is sound. NestJS + Next.js + Prisma + Turborepo is a competent choice. Controller/service/repository separation is followed. Shared API contracts via a package is the right pattern. But the topic color/icon system is hardcoded against closed data, global CSS has no encapsulation, test coverage is thin for the claimed complexity, and the homepage has a first-class regression baked into its architecture.

### Issues

**Topic color and icon system will silently degrade for any new topic**
The per-topic accent color system is implemented as CSS data-attribute selectors with hardcoded slug values:

- `collection.css` — `.topicCollectionItem[data-topic='democracy']` through `local-community` (7 rules)
- `pages.css` — `.heroPosterIssueLink[data-topic='democracy']` through `local-community` (7 rules)
- `components/topic-summary.tsx:22-30` — `TOPIC_ICON_MAP` keyed by slug (7 entries)

Topic admin now allows creating new topics. A new topic will render in Issues, Articles, Actions, and Events with zero icon and the fallback amber accent color — identical to every other card. There is no graceful degradation path. The system isn't designed for the data model it now supports.

**90 source files, 58 test files — ratio is thin given claimed scope**
The project has 90 API source files and 58 test files total across both web and API. Given that Phases 4 through 12 explicitly built out integration testing infrastructure, controller tests, and frontend tests, and given that the stated goal is portfolio engineering credibility, this ratio should be higher. The test count doesn't indicate what's covered versus what isn't, but it suggests large surfaces with no automated validation.

**Global CSS with no encapsulation, 1517-line admin.css**
All styles across the project are global CSS files imported at the layout level. `admin.css` is 1517 lines. There are no CSS modules for components. Class names like `.metaText`, `.collectionItem`, `.detailContent`, `.primaryCTA` are globally scoped. There is no mechanism preventing a class added in `admin.css` from accidentally matching public routes or vice versa. The `tokens.css` file defines design tokens, but `admin.css` still has hardcoded hex values: `color: #425466`, `color: #171717` used in multiple places that bypass the token system.

**`secondaryCTA` has a hardcoded light-mode color**
`pages.css:429-430`:

```css
border: 1px solid #171717;
color: #171717;
```

This is the base `secondaryCTA` before the `.publicShell` override. On the dark background it would render as an invisible black button on a black background — the `.publicShell` override saves it, but the base definition is wrong and will cause issues if the component is ever used outside `.publicShell`.

**`TopicService.getTopicDetail` fans out to two external services**
`topic.service.ts:28-33` calls both `articleService.getArticlesForTopic` and `actionService.getActionsForTopic` via `Promise.all`. This makes the `TopicService` dependent on two other domain services. If either fails, the topic detail fails entirely. The fan-out would be more robust and more cohesive at the repository layer with a single query, but more importantly, this creates a tight coupling that makes the topic service harder to test in isolation and harder to refactor.

**Inline SVG data URI grain texture in CSS**
`pages.css:24-31` — the grain texture is generated by an inline SVG data URI in a CSS `background-image`. This is technically valid but it's a non-standard technique that is difficult to maintain, hard to inspect in DevTools, and may behave inconsistently across rendering engines. If the grain is a deliberate brand element, it should be a real asset file. If it's not important enough to be a real asset, it's not important enough to keep.

**`topicCollectionItemTilt` applies rotation on hover in a dense grid**
`collection.css` — `.topicCollectionItemTilt:hover` applies `transform: translate(-3px, -4px) rotate(0.6deg)`. In a 2-column grid at narrow widths, a hovered card physically overlaps adjacent cards. The `z-index: 1` on hover prevents content clipping but the layout shift is perceptible and reads as jitter rather than polish at smaller viewport sizes.

**The `aria-label="Search"` workaround in nav**
`layout.tsx:38` — the Search nav link uses `aria-label="Search"` to display an icon instead of text. The CSS then targets `a[aria-label='Search']` to style it differently from other nav links (`layout.css:136-145`). Using an ARIA attribute as a CSS selector hook is an anti-pattern: ARIA attributes are for accessibility semantics, not for style targeting. Add a CSS class.

## What Is Actually Good

- **The homepage hero headline.** "Which one is yours?" is direct, provocative, and product-aligned. The issue roll below it with Playfair Display at 2.8rem is the strongest visual moment on the site.
- **The Events filter surface.** Screenshot 03 shows a well-designed filter interface: region/city selectors, date pickers, issue filter pills. This is more polished than most civic project demos.
- **The shared API contracts package.** Using a shared `@signal-fire/api-contracts` package for response types shared between API and web is the right pattern and executed correctly.
- **The governance documentation.** The `progress.md` file alone is genuinely impressive — phase-by-phase tracking with definition-of-done criteria, scope boundaries, and architectural decisions. This shows disciplined process.
- **Breadcrumb and footer navigation.** The `detailBreadcrumbLink` (← Climate) and "Browse more actions" footer CTAs eliminate dead ends on detail pages. These were clearly added deliberately and they work.
- **Per-topic accent colors on the Issues grid.** The `data-topic` attribute approach for left-border color differentiation is elegant and easy to extend — for the hardcoded topic set.
- **The NestJS architecture.** Controller/service/repository separation is consistent. The `NotFoundException` pattern at the controller layer is correct. The `public-content.mapper` for response shaping is a clean separation.
